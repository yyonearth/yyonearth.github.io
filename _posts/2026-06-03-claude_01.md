做数据这行久了，最常被问的一句话是：「能不能让不会写 SQL 的人也能查数据？」

过去这是个伪需求——自然语言太模糊，SQL 又要求精确。但大模型成熟之后，「自然语言转 SQL」（NL2SQL）从 demo 真正变成了可以放进业务的东西。这篇就以我用 **Claude** 做 NL2SQL 的经历，聊几个真正影响成败的工程点。不谈玄学，只谈落地时绕不开的坑。

## 一、模型不知道你的表，所以 schema 注入是第一关

新手最容易犯的错，是直接把用户问题丢给模型让它写 SQL。结果模型会一本正经地编出根本不存在的表名和字段。

模型不可能知道你的数据库长什么样。所以真正的第一步，是把**表结构（schema）作为上下文喂进去**：表名、字段名、字段类型、关键的外键关系，必要时再加上几行示例数据。

这里有个现实矛盾：库大了，schema 可能有几百张表，全塞进 prompt 既超 context 又烧 token。我的做法是先做一轮**表召回**——根据用户问题用向量检索或关键词匹配，挑出最相关的少数几张表，再把这几张的 schema 注入。先选表，再生成 SQL，准确率和成本都明显更好。

> 一句话原则：模型生成的质量，上限取决于你给它的上下文质量。NL2SQL 八成的功夫在 schema 怎么给，而不是 prompt 怎么写得花哨。
{: .prompt-tip }

## 二、让模型输出结构化结果，而不是一段话

模型默认是「聊天」的，它很愿意在 SQL 前后加一堆解释。可你的后端要的是一段能直接执行的 SQL，不是散文。

Claude 支持**结构化输出**——你可以强制它按你给的 JSON Schema 返回，比如：

```json
{
  "sql": "SELECT ...",
  "explanation": "这条查询统计了...",
  "tables_used": ["orders", "users"]
}
```

这样后端拿到的就是干净的字段，不用再写正则去从一段话里抠 SQL。`sql` 直接执行，`explanation` 给用户看，`tables_used` 还能用来做审计和调试。**能用结构化输出的地方，就不要让模型自由发挥**，这是工程稳定性的关键一步。

## 三、模型会生成「看起来对」的 SQL，所以必须验证

这是最容易被低估的一点：模型生成的 SQL 语法通常没问题，**但语义不一定对**——它可能 join 错了表、用错了聚合、或者悄悄全表扫描。

我的流水线里，模型出 SQL 之后还有几道关卡：

- **语法校验**：先用 `EXPLAIN` 跑一遍，不执行也能查出语法/字段错误
- **权限与安全**：只允许 `SELECT`，拦截任何 `DELETE` / `UPDATE` / `DROP`；这是底线
- **代价检查**：看执行计划，对全表扫描、笛卡尔积这类做拦截或警告
- **结果回看**：把查询结果摘要再发给模型，让它判断「这是否回答了用户的问题」

把模型当成一个**很聪明但需要被复核的实习生**，而不是一个能直接上生产的引擎。这个心态决定了你系统的可靠性。

## 四、省钱省延迟：缓存与分层

真实业务里成本和延迟是要命的。两个最有效的手段：

**Prompt 缓存。** schema、系统提示这些内容每次请求几乎不变，却占了 prompt 的大头。Claude 的 prompt caching 能把这部分缓存住，重复请求时不重复计费、还更快。把「稳定不变的上下文」和「每次都变的用户问题」分开摆放，让前者尽量命中缓存。

**模型分层。** 不是每个请求都要用最强的模型。简单的意图识别、表召回，用更小更快的模型（如 Haiku）就够；只有真正生成复杂 SQL 时才上 Opus / Sonnet。按难度分层，体验和成本能兼顾。

## 五、没有评测，就是在凭感觉

最后一点，也是从 demo 走向产品最大的分水岭：**你得有一套评测集**。

准备一批「自然语言问题 → 正确 SQL（或正确结果）」的样例，每次改 prompt、换模型、调流程，都跑一遍，看准确率是涨是跌。没有这套东西，你所有的「优化」都只是自我感觉良好，根本不知道某次改动到底变好了还是变差了。

> AI 工程和传统软件工程最大的区别，是它的行为是概率性的。所以「可测、可回归」比「这次能跑」重要得多。
{: .prompt-tip }

## 小结

把大模型用进真实业务，难点从来不在「让它生成」，而在它周围那一圈工程：

1. **上下文**——schema 注入 + 表召回，决定生成质量上限
2. **结构化**——强制 JSON 输出，让结果可被程序消费
3. **验证**——语法、权限、代价、语义层层把关
4. **成本**——prompt 缓存 + 模型分层
5. **评测**——有评测集才有真正的迭代

模型只是其中一环。**能不能落地，取决于你给它搭了多结实的脚手架。** 后面我会再写一篇，聊聊把这套思路用到「智能外呼」这类更复杂的多轮场景时，又会遇到哪些新问题。
