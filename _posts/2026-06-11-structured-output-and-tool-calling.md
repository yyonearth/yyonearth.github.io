上一篇聊 NL2SQL 时我说过一句话：模型生成的质量，上限取决于你给它的上下文质量。这篇换个角度——光让模型「答得对」还不够，在工程里更难的一关是让它**答得能用**。

什么叫能用？就是模型吐出来的东西能直接喂给下一段代码，而不是先经过你一通正则、字符串切割、`try/except` 才能勉强解析。聊天框里模型说得头头是道，可一旦你想把这段输出接进业务流程，就会发现自由文本是工程的天敌。这篇就讲怎么把模型的嘴管住——**结构化输出**和**工具调用**。

## 一、为什么「能聊」不等于「能用」

举我自己的例子。做信用卡数据洞察那个 App，链路是「自然语言 → SQL → 图表」。中间有一步：模型不仅要写出 SQL，还得告诉前端这数据该画成柱状图、折线图还是饼图。

最早我图省事，让模型直接用自然语言回答：「这个查询适合用折线图展示月度趋势」。然后我在后端写正则去抠「折线图」三个字。结果第二天模型回我「建议采用折线图（line chart）来呈现」，正则没匹配上，前端直接白屏。

问题不在模型答错了，它答得很对。问题在于**自由文本的格式是不稳定的**，而代码需要的是稳定的契约。你没法对一段随时会换措辞的中文做可靠的程序化解析。

> 一句话原则：只要下游是代码而不是人，就不要让模型输出自由文本。人能读懂的措辞变化，恰恰是代码最怕的东西。

## 二、第一招：用 JSON Schema 把输出格式钉死

最直接的办法，是要求模型输出 JSON，并且给它一个明确的 schema。比如上面那个场景，我要的不是一段话，而是这么个结构：

```json
{
  "sql": "SELECT month, COUNT(*) FROM ... GROUP BY month",
  "chart_type": "line",
  "x_field": "month",
  "y_field": "count"
}
```

这里有个关键认知的升级：从**「在 prompt 里求模型」**到**「在 API 层强制模型」**。

早期大家都是在提示词里写「请只返回 JSON，不要带任何解释」。这能用，但不可靠——模型时不时会手痒加一句「好的，这是您要的 JSON：」，或者用 ```` ```json ```` 把它包起来，你还得再剥一层。

现在主流的模型 API 都支持**结构化输出 / JSON mode**：你把 JSON Schema 作为参数传进去，模型被约束在解码阶段就只能生成符合 schema 的 token。`chart_type` 你用枚举限定成 `["line", "bar", "pie"]`，它就吐不出第四种值。这不是靠提示词「请它配合」，而是从机制上让它没法越界。

能用 API 层的 schema 约束，就别只靠 prompt 里的口头约定。前者是栏杆，后者是路标——路标可以无视，栏杆不行。

## 三、工具调用：让模型「选动作」，而不是你去猜它的话

结构化输出解决的是「输出长什么样」，工具调用（Function Calling）解决的是更进一步的问题：**让模型自己决定调用哪个函数、传什么参数**。

还是那个 App，用户的问题其实有好几类：有的是查数据（要走 SQL），有的是问「这个指标怎么算的」（要走文档检索），有的只是闲聊。早期我用一个大 prompt 让模型自己判断意图再回答，逻辑全糊在提示词里，加一种新意图就要改一大段，且经常顾此失彼。

换成工具调用之后，思路干净了很多。我把每个动作定义成一个「工具」，告诉模型它有这些工具可用：

```python
tools = [
    {
        "name": "query_database",
        "description": "当用户想查询具体数据时调用",
        "parameters": {
            "type": "object",
            "properties": {
                "sql": {"type": "string"}
            },
            "required": ["sql"]
        }
    },
    {
        "name": "search_docs",
        "description": "当用户询问指标定义、口径解释时调用",
        "parameters": { "...": "..." }
    }
]
```

模型不再用自然语言告诉我「我觉得应该查数据库」，而是直接返回一个结构化的调用意图：调 `query_database`，参数 `sql` 是某段查询。我的代码拿到的是函数名加参数，直接 `dispatch` 过去执行就行——**意图识别和参数提取这两件事被合并成了一次结构化输出**。

> 一句话原则：工具调用的本质，是把「模型用文字描述它想干什么」变成「模型直接告诉你调哪个函数、传什么参」。前者要你猜，后者你只管执行。

## 四、模型一定会偶尔违约，所以验证和重试是标配

说了这么多结构化的好处，泼盆冷水：**即便有 schema 约束，模型仍然会偶尔给你不合法的输出**。字段缺失、枚举值越界、JSON 截断（尤其是 token 超了被切断）、SQL 语法错……概率不高，但在生产里跑久了一定会撞上。

所以结构化输出从来不是「约束完就万事大吉」，它必须配一套兜底：

1. **解析即校验**。拿到输出第一件事就是按 schema 校验一遍（Python 里我习惯用 Pydantic，模型返回直接灌进 model，不合法立刻抛错）。别等到下游用的时候才发现字段是 `None`。
2. **失败就重试，并且把错误回喂给模型**。第一次返回的 JSON 缺了 `y_field`，第二次请求时我会把「上次你漏了 y_field，请补全」连同原问题一起发回去。带着具体错误的重试，比干重试有效得多。
3. **设重试上限 + 降级路径**。重试两三次还不行，就别死磕了——要么返回一个对用户友好的「没太理解，换个说法试试」，要么降级到一条预设的安全查询。绝不能让一次模型抽风把整个请求挂死。

我后端那层 NL2SQL 服务，真正写「调模型」的代码可能就十几行，剩下大半都是在做这件事：校验、重试、容错、降级。**这部分代码的厚度，基本决定了你这个 AI 功能在生产里能不能扛得住。**

> 一句话原则：把模型当成一个聪明但偶尔不靠谱的下游服务。你不会因为某个 HTTP 接口大概率成功就不写错误处理，调模型也一样。

## 五、小结

把这几点串起来，其实是一条主线——**让模型的输出从「给人看的话」变成「给代码用的数据」**：

- 下游是代码，就别让模型输出自由文本；
- 能用 API 层的 JSON Schema 强约束，就别只靠 prompt 里口头求它；
- 用工具调用把「意图判断 + 参数提取」收敛成一次结构化输出；
- 永远假设模型会偶尔违约，校验、重试、降级一个都不能少。

大模型落地这一年我最大的体感是：真正拉开差距的，往往不是 prompt 写得多花哨，而是你把模型这个「不稳定组件」工程化包装得有多扎实。结构化输出和工具调用，就是这层包装最基础、也最值钱的两块砖。
